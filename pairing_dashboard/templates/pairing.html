Не путать с «device» в OpenClaw.
Здесь — ноды (приложения / headless-клиенты в роли node), которые просят доступ к шлюзу. Сюда не попадает одобрение операторского device (браузер Control UI, CLI, ключ сервиса pairing_ws_device.json на поде). Тот список — в openclaw devices pending / веб-UI шлюза (порт 18789), одобрение: openclaw devices approve <requestId>.

Два paired «operator» в devices — часто норма: один identity у браузера, второй у бэкенда этой страницы (:8081) на диске /workspace/.guildclaw/pairing_ws_device.json — разные ключи, оба валидны.

Почему с телефона пусто: по доке OpenClaw в эту очередь попадают только клиенты, которые по WebSocket шлюза вызывают метод node.pair.request. Обычный вход в Control UI / pairing как operator (device) туда не пишется. В нодовом приложении укажите тот же шлюз, что и у этого пода: на RunPod обычно wss://ваш-pod-id-18789.proxy.runpod.net, токен — как OPENCLAW_WEB_PASSWORD (или что вы задали в приложении). На поде проверка: openclaw nodes pending — список должен совпадать с таблицей ниже (если там пусто, и здесь будет пусто).
